## What this project is

TokoMo is an npm package (`@ds-mo/tokens`) that ships **design tokens** as:

- CSS custom properties (the primary deliverable тАФ one file per category + a combined index)
- Light/dark theme files (`dist/themes/light.css`, `dist/themes/dark.css`)
- A machine-readable JSON blob (`dist/tokens.json` + per-category JSON files)
- TypeScript constants for all token names (`dist/index.mjs` / `.cjs` / `.d.ts`)
- Reset and global utility CSS

It's the **foundation** of the ds-mo design-system trilogy: `@ds-mo/tokens` тЖТ `@ds-mo/icons` тЖТ `@ds-mo/ui` (CompoMo). TokoMo is Figma-first тАФ raw token JSON is exported from Figma variables and dropped into `src/`, then build scripts generate the distributable artifacts.

## Token index тАФ instant lookup for AI agents

`dist/tokens-index.json` (generated by `npm run build`) groups every token by semantic subcategory. A single `Read` of this file тАФ no grep needed.

**Top-level keys:**

Structure is one level deep тАФ top-level category, then subcategory (colors only), then tokens:

```
colors.reference  тЖТ  --color-reference-*   (raw palette, matches src/json/colors/reference/)
colors.semantic   тЖТ  all other --color-*   (matches src/json/colors/semantic/)
colors.data       тЖТ  --color-data-*        (data-viz palettes, matches src/json/colors/data/)
dimensions        тЖТ  --dimension-*
typography        тЖТ  --typography-*
effects           тЖТ  --effect-*            (animation, motion, blur, shadow, elevation)
```

**Lookup patterns:**

```js
// All semantic colors
const semanticColors = tokensIndex.colors.semantic;

// All reference colors
const referenceColors = tokensIndex.colors.reference;

// All effect tokens
const effects = tokensIndex.effects;

// Exact value of a single token
const value = tokensIndex.effects['--effect-motion-short-2'].$value;

// Top-level category names
const categories = tokensIndex._meta.categories;
```

**Package export:** `import tokensIndex from '@ds-mo/tokens/json/index'`

## Commands

```bash
npm run build          # Full build тАФ CSS + JSON + TypeScript
npm run build:colors   # Color tokens only (fast iteration)
npm run build:docs     # Rebuild docs/index.html (GH Pages browser)
npm run dev            # Watch mode тАФ rebuilds on src changes
npm run clean          # Remove dist/
```

No separate test/lint commands тАФ validation is done by the Build workflow on every PR (it re-runs the build and asserts `src/` was not mutated).

---

## Build pipeline (what `npm run build` does)

1. **Clean** тАФ nuke `dist/`, recreate subdirs (`dist/themes/`, `dist/json/`)
2. **Generate CSS** тАФ each `generate-*-tokens.mjs` reads `src/json/**` and writes **`src/*.css`** (committed source)
3. **Copy to dist** тАФ `scripts/build.mjs` copies `src/*.css` and `src/themes/` into `dist/`
4. **Generate JSON** (`generate-json-tokens.mjs`) тАФ parses **`src/*.css`** custom properties тЖТ `dist/tokens.json`, `dist/json/*.json`, `dist/tokens-index.json`
5. **Generate TypeScript** (`generate-ts-constants.mjs`) тАФ emits `dist/index.mjs`, `dist/index.cjs`, `dist/index.d.ts`

## Theming

Light/dark theming is **CSS-only** тАФ no JavaScript. Consuming apps toggle the `data-theme` attribute on `:root` (or a container element):

```html
<html data-theme="dark">
```

All token values are defined as CSS custom properties. **Color** light/dark values live in `src/colors.css` under `:root` and `:root[data-theme="dark"]`. `src/themes/light.css` and `dark.css` only set `color-scheme` for native form controls.

## Adding or updating tokens

### From a Figma export

1. Export Figma variables as JSON (one file per collection: colors, dimensions, typography, effects).
2. Drop the JSON files into `src/json/` replacing the existing files.
3. Run `npm run build` to regenerate `dist/`.
4. Verify the token browser: `npm run build:docs`, then open `docs/index.html`.

### Editing a token directly

1. Edit the JSON file in `src/json/` for that category.
2. Run `npm run build` (or `npm run build:colors` for color-only changes).
3. Do **not** edit generated CSS in `dist/` тАФ your changes will be overwritten on the next build.

### Adding a new token category

1. Create `src/json/<category>.json` with the token definitions.
2. Write `scripts/generate-<category>-tokens.mjs` following the pattern of existing generators.
3. Add the new script to the build sequence in `scripts/build.mjs`.
4. Update `scripts/generate-json-tokens.mjs` to include the new category in the merged output.
5. Update `package.json` `exports` with the new `dist/<category>.css` entry.
6. Update `src/index.css` to `@import` the new category.

## Commit & PR conventions

**Conventional Commits**, enforced by `.github/workflows/pr-title.yml`:

```
<type>(<optional-scope>): <lowercase subject>

types: feat | fix | perf | revert | docs | style | refactor | test | build | ci | chore
```

Subject must **start with a lowercase letter** (workflow enforced). Scope is optional тАФ common ones here: `colors`, `docs`, `build`.

**Version-bumping types** (trigger a release PR via release-please):
- `feat:` тЖТ minor bump
- `fix:` / `perf:` тЖТ patch bump
- `feat!:` or `BREAKING CHANGE:` footer тЖТ major bump

Breaking token renames require a **major** semver bump (`@ds-mo/tokens` is post-2.0).
- `ci:` / `chore:` / `build:` / `test:` / `style:` / `docs:` / `refactor:` тЖТ **do not trigger a release** (most hidden in changelog; `docs` is visible)

See `release-please-config.json` for the type тЖТ changelog section mapping.

**Branch naming:** `type/short-kebab-description` (e.g. `feat/add-motion-tokens`, `ci/add-release-workflow`, `docs/agent-onboarding`).

**PR flow:** always via feature branch + PR to `main`. Direct pushes to `main` are blocked.

## Versioning

Breaking token renames require a **major** semver bump.

Current version lives in three places тАФ keep them in sync for releases not driven by release-please:
- `package.json` `"version"`
- `.release-please-manifest.json` `"."`
- `README.md` description (if it mentions token counts)

Release-please handles all three automatically when it opens a release PR.

## Release flow

**Automated path (normal case):**

1. Land a `feat:` or `fix:` commit on `main` via PR.
2. `release-please.yml` fires тЖТ opens (or updates) a release PR that bumps `package.json`, updates `CHANGELOG.md`, and updates `.release-please-manifest.json`.
3. Review and merge the release PR.
4. Release Please tags `vX.Y.Z`, creates the GitHub Release, and the `publish` job in the same workflow publishes to npm with `--provenance` via **OIDC Trusted Publisher** (no long-lived `NPM_TOKEN` тАФ configured in npm under Package Settings тЖТ Trusted Publishers).

**Forcing a specific version (`Release-As:` escape hatch):**

Push an empty commit with a `Release-As: X.Y.Z` trailer in the commit message body to `main`:

```bash
git commit --allow-empty -m "chore: release as X.Y.Z

Release-As: X.Y.Z"
```

Release-please will open a release PR at that exact version. Useful when only `ci:`/`chore:` commits have accumulated and you want to cut a release.

**Merge strategy:** use "Create a merge commit" (not squash) when merging a `Release-As:` commit so the trailer survives. If squash is enforced, paste `Release-As: X.Y.Z` into the squash commit message body manually.

**Never** run `npm publish` manually for a normal release тАФ it bypasses provenance and skips the tag/release/changelog dance.

## Things not to do

- **Do not edit `dist/`** тАФ it's generated. Edit `src/` or scripts, then run `npm run build`.
- **Do not edit `docs/index.html`** тАФ regenerate with `npm run build:docs`.
- **Do not hand-bump `package.json` version** during normal work тАФ let release-please do it.
- **Do not `git push` to `main`** тАФ always branch + PR.
- **Do not delete a token from `src/json/` without explicit user confirmation** тАФ even if a Figma re-export omits it; it might be a Figma filter accident.
- **Do not commit `NPM_TOKEN` or any npm auth** тАФ publishing uses OIDC, no secrets required.
- **Do not skip the npm upgrade step** in the publish job тАФ Node 20 ships with npm 10.x which cannot complete OIDC auth; Trusted Publisher requires npm тЙе 11.5.1. Pin `npm@^11.5.1` (not `@latest` тАФ npm 12+ requires Node 22+).
